The thesis
Not just an AI assistant.
A disciplined workbench.
You describe an outcome. The system handles the rest. Six moves. One spine.

1. Decompose

The outcome is split into research, design, implementation, verification, documentation, follow-up.

2. Route

Each piece goes to the right model. Cost. Quality. Privacy. Latency. Local or hosted.

3. Verify

Deterministic tools prove what happened. Every claim has an artifact. No vibes.

4. Document

Every failure becomes a regression guard. Knowledge flows back into a durable wiki.

5. Approve

Human approval is reserved for judgment calls, risk boundaries, and product taste.

6. Follow-up

The next commission starts smarter. The system gets sharper while you sleep.

How the router decides

Cost / quality / privacy / latency. Reflection is hard, that's Opus. Write to memory is deterministic, that's a tool, no model. Voice reply must be fast, that's Haiku + local TTS. The router never asks the smart model to do a dumb job.

Every claim has an artifact

  • The path posts in Discord text. You can check it
  • "Three lessons extracted" is now a thing on disk, not a vibe

Every failure becomes a regression guard

  • If the agent ever tries to write garbage there, an invariant catches it
  • That invariant came from a real failure, not a hypothesis
  • Tomorrow the same class of mistake costs zero
  • The system gets less stupid every week
No artifact, no commission. The contract is the file path that appears, not the warm feeling that something happened.

Without the workbench

  • Inbox: read every message, decide every reply
  • PRs: review every diff, approve every merge
  • Calendar: ten conflicts a week, you resolve all of them
  • Memory: Notion docs nobody re-reads
  • End of week: tired, behind, the same things to do Monday

With the workbench

  • Inbox: agent triages, you read 8 things instead of 78
  • PRs: regression guard runs first; you approve judgment, not typos
  • Calendar: ten conflicts handled, escalated only when there's a person to choose between
  • Memory: the wiki gets sharper every commission
  • End of week: same hours worked, twice the things shipped
Same hours. Different outputs. The advantage of small is focus, not headcount.
Before you start
Four traps. One uncomfortable truth.

1. Full-system thinking

Wanting the whole stack before you've run a single agent through a real workflow. Start with one card, one outcome, one artifact.

2. Better-model thinking

"Maybe Opus 5 will fix it." A stronger LLM won't clean your data, fix your retrieval, or design your workflow. The bottleneck is the OS, not the kernel.

3. Skipping oversight

"Set it and forget it" agents go feral inside a week. Read every artifact for the first month. Earn the trust before you delegate it.

4. Wrong-tool fit

Enterprise orchestration platforms for solo problems. A markdown file and a shell loop beat a SaaS tier you'll never use.

The uncomfortable truth: the first two weeks feel slower. You're learning to issue commissions instead of vibes. If I don't say that out loud, the serious people will discover it alone and trust me less.

Closing
Build the machinery.
In 10 years, my work is less about manually stitching tools together and more about shaping reliable loops: defining what good means, giving the system enough context, watching the evidence, deciding which problems are worth pointing it at.

Small businesses, creators, researchers, solo operators using systems like this to build software, run audits, analyse markets, automate operations, and maintain institutional memory without needing a large staff or a huge SaaS budget.
